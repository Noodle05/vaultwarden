# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Test

```bash
# Build (must enable one DB backend)
cargo build --features sqlite
cargo build --features postgresql
cargo build --features mysql

# Run with all features you need
cargo run --features sqlite

# Run tests (uses Docker for DB backends)
cargo test --features sqlite

# Linting
cargo clippy --features sqlite -- -Dwarnings
cargo fmt -- --check

# Generate admin token hash
cargo run -- hash
```

DB backend features are mutually exclusive in practice: `sqlite`, `postgresql`, `mysql`. `sqlite` bundles SQLite statically; `sqlite_system` links dynamically. S3 attachment support via `s3` feature.

Rust toolchain is pinned in `rust-toolchain.toml` (1.95.0, MSRV 1.93.0). CI enforces `-Dwarnings`.

## Architecture

**Framework:** Rocket 0.5 with `rocket_ws` for WebSocket. Tokio async runtime. Diesel ORM with `diesel_migrations` for schema management.

**Config:** `src/config.rs` uses a `make_config!` macro that generates typed accessors from a struct definition. Config is loaded from env vars (`.env` file) and/or `DATA_FOLDER/config.json`. The `CONFIG` global is a `LazyLock<Config>`.

**Database:** `src/db/mod.rs` — `DbConnInner` is a `diesel::MultiConnection` enum with one variant per DB backend. `db::run_blocking(f)` wraps synchronous Diesel calls in `spawn_blocking`. Connection pool is `DbPool` (r2d2). DB models live in `src/db/models/` — each file defines a struct + Diesel query methods. `src/db/schema.rs` is generated by Diesel.

**API routing:** `src/main.rs` `launch_rocket()` mounts:
- `/` → `api::web_routes()` — static web vault files
- `/api` → `api::core_routes()` — Bitwarden API (accounts, ciphers, folders, sends, orgs, emergency-access)
- `/admin` → `api::admin_routes()`
- `/events` → `api::core_events_routes()` — org event logs
- `/identity` → `api::identity_routes()` — OAuth/OIDC identity provider
- `/icons` → `api::icons_routes()` — website icons proxy/cache
- `/notifications` → `api::notifications_routes()` — WebSocket hub (`/hub`, `/anonymous-hub`)

**Auth:** `src/auth.rs` — JWT-based (HMAC or RSA). Request guards: `Headers` (general auth), `AdminHeaders` (admin panel), `WsAccessTokenHeader` (WebSocket), `ClientIp`. Org user enrollment/confirmation via `InviteJwt`.

**Notifications (WebSocket + Push):** `src/api/notifications.rs` contains the `WebSocketUsers` hub — a `DashMap<String, Vec<(Uuid, Sender)>>` keyed by user UUID. Each open client tab gets an mpsc `Sender`. When a mutation occurs, the corresponding `send_*_update` method fans out a MsgPack-encoded SignalR binary message to all that user's connected clients AND triggers a push notification via `src/api/push.rs` (Bitwarden cloud push relay for mobile). The `UpdateType` enum (line 620) defines all notification types.

**Migrations:** `migrations/{sqlite,mysql,postgresql}/` — one directory per backend. Schema managed by `diesel_migrations` embedded migration macros.

**Scheduled jobs:** `src/main.rs` `schedule_jobs()` — runs in a dedicated thread using `job_scheduler_ng`. Jobs: send purge, trash purge, incomplete 2FA notifications, emergency access timeouts/reminders, auth request purge, Duo context purge, event cleanup, SSO auth purge.

**Key dependencies:** `rocket` 0.5, `diesel` 2.x, `rmpv` (MsgPack), `openssl`/`aws-lc-rs` (TLS), `opendal` (S3), `hickory-resolver` (DNS), `openidconnect` (OIDC), `argon2` (password hashing).

**Active work:** See `docs/ha-notification-design.md` for the HA multi-pod notification design (PostgreSQL triggers + sidecar + internal REST API). Not yet implemented.
